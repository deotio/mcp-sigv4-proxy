# mcp-sigv4-proxy v0.4.4 ‚Äî Implementation Plan

Six targeted fixes for the fundamental flaws identified during debugging.
Changes are ordered by impact; each is independent and can be reviewed separately.

---

## Fix 1 ‚Äî Signing timeout (`processLine` + `warmBackend`)

**Problem**: `signer.sign()` resolves credentials lazily via `fromNodeProviderChain()`. With
expired SSO it silently attempts a network token refresh with no timeout. Every call blocks for
~60 s before throwing. Since `processLine` is called serially, three list requests (tools,
resources, prompts) take up to 180 s before all fail.

**Solution**: Wrap every `signer.sign()` call in a `Promise.race` against a short timeout.

### New constant
```typescript
const DEFAULT_SIGN_TIMEOUT_MS = 10_000; // 10 s ‚Äî fail fast if credentials are unavailable
```

Add opt-in env var `MCP_SIGN_TIMEOUT` (seconds, same pattern as `MCP_TIMEOUT`). Validate and
log in `validateEnv`; store on `ProxyConfig` as `signTimeoutMs`.

### New function (place after `createSigner`)
```typescript
export async function signWithTimeout(
  signer: SignatureV4,
  request: HttpRequest,
  timeoutMs: number,
): Promise<Awaited<ReturnType<SignatureV4['sign']>>> {
  let timer: ReturnType<typeof setTimeout>;
  const timeout = new Promise<never>((_, reject) => {
    timer = setTimeout(
      () => reject(new Error(`credential resolution timed out after ${timeoutMs}ms ‚Äî ` +
        'run: aws sso login --profile <your-profile>')),
      timeoutMs,
    );
  });
  try {
    return await Promise.race([signer.sign(request), timeout]);
  } finally {
    clearTimeout(timer!);
  }
}
```

### Changes to `processLine`
Replace:
```typescript
signed = await signer.sign(request);
```
With:
```typescript
signed = await signWithTimeout(signer, request, config.signTimeoutMs);
```

### Changes to `warmBackend`
Replace both `await signer.sign(request)` calls (one in the initialize loop, one in the
`Promise.all` prefetch) with `signWithTimeout(signer, request, config.signTimeoutMs)`.

### Tests to add (unit, `signWithTimeout` suite)
- `resolves with signed request when sign completes within timeout`
- `rejects with timeout error when sign exceeds timeout`
- `clears the timer on success (no leak)`

### Tests to update
- `emits JSON-RPC error when signing fails` ‚Äî the error message now includes the timeout hint;
  update assertion.

## Fix 2 ‚Äî No error response for notifications (`processLine`)

**Problem**: JSON-RPC notifications have no `id`. `parseInputLine` returns `requestId = null`.
When signing or network fails for a notification, the proxy writes
`{"jsonrpc":"2.0","id":null,"error":{...}}` to stdout. Claude Code's MCP SDK Zod schema rejects
this combination, which dropped the connection.

**Solution**: Track whether the input is a notification and skip the error response if so.

### Change to `parseInputLine`
Return an additional `isNotification` boolean:
```typescript
export function parseInputLine(
  line: string,
): { body: string; requestId: unknown; isNotification: boolean } | null {
  // ... (existing logic unchanged until return)
  const requestId = (parsed as Record<string, unknown>).id ?? null;
  const isNotification = !('id' in (parsed as Record<string, unknown>));
  return { body, requestId, isNotification };
}
```

### Change to `processLine`
After each error path that writes to stdout, guard with:
```typescript
if (!input.isNotification) {
  process.stdout.write(JSON.stringify({ jsonrpc: '2.0', id: requestId, error: { ... } }) + '\n');
}
```

Apply to all three error write sites: signing failure, fetch failure, timeout.

### Change to `handleWarmLine`
`handleWarmLine` already uses `parseInputLine` indirectly; the `tryWarmResponse` path checks for
a cached result and writes it. For notifications this is fine (no cached result ‚Üí falls through).
No change needed there. But add guard in the `initialize` intercept path:
```typescript
// initialize is always a request (has id), so this is safe ‚Äî but be explicit:
if (input.isNotification) return false; // notifications are never intercepted
```

### Tests to add
- `parseInputLine: marks notification (no id field) as isNotification=true`
- `parseInputLine: marks request with id:null as isNotification=false`
- `processLine: does not write to stdout for notification when signing fails`
- `processLine: does not write to stdout for notification on network error`

### Tests to update
- `notification errors` integration test: verify NO line is written to stdout (currently asserts
  an error is written ‚Äî this changes behavior).

## Fix 3 ‚Äî Add `Accept` header (`buildHttpRequest`)

**Problem**: MCP Streamable HTTP requires `Accept: application/json, text/event-stream`.
Without it the AgentCore server may respond with only one content type or return 406.
The warm-up error handler already has a `406 ‚Üí possibly missing Accept header` hint,
confirming this is a known risk.

**Solution**: Add the header unconditionally.

### Change to `buildHttpRequest`
```typescript
headers: {
  'Content-Type': 'application/json',
  'Accept': 'application/json, text/event-stream',
  'Content-Length': String(Buffer.byteLength(body)),
  host: url.hostname,
},
```

### Tests to update
- All `buildHttpRequest` tests that assert on the headers object: add `Accept` assertion.
- `processLine` tests that capture the outgoing `fetch` call headers: verify `Accept` is present.

## Fix 4 ‚Äî Fail fast on credential errors in `warmBackend`

**Problem**: When SSO credentials are expired the warm-up catch block retries up to 5 times
with `10 s, 20 s, 40 s, 80 s, 160 s` delays (‚â•5 min total). Expired tokens don't fix
themselves. The warm-up grinds silently for minutes, the synthetic `initialize` has already
misled Claude Code, and real-time requests are also failing.

**Solution**: Detect credential errors and return `false` immediately on the first attempt.

### Change to `warmBackend` (initialize retry loop catch block)
```typescript
} catch (err) {
  const isCredentialError =
    err instanceof Error &&
    (err.message.includes('credential') ||
      err.message.includes('Could not load credentials') ||
      err.message.includes('profile') ||
      err.message.includes('timed out')); // catches signWithTimeout errors

  if (isCredentialError) {
    // No point retrying ‚Äî expired tokens won't fix themselves.
    log('ERROR', `warm: credential error ‚Äî aborting warm-up. ` +
      `Run: aws sso login --profile <your-profile>\n  Details: ${err}`);
    return false;
  }

  if (attempt < config.warmRetries) {
    const delay = config.warmRetryDelayMs * Math.pow(2, attempt);
    log('WARNING', `warm: initialize error (${err}), retrying in ${delay}ms (${attempt + 1}/${config.warmRetries})`);
    await sleep(Math.min(delay, deadline - Date.now()));
    continue;
  }
  log('ERROR', `warm: initialize failed after ${config.warmRetries} retries: ${err}`);
  return false;
}
```

Also apply to the `Promise.all` prefetch catch blocks (each `catch(err)` inside the map): if
credential error, log ERROR and return (the `Promise.all` will still complete, just without that
cache entry).

### Tests to add
- `warmBackend: returns false immediately on credential error without retrying`
- `warmBackend: does not retry prefetch on credential error`

### Tests to update
- Existing retry tests: ensure they still fire for non-credential errors (e.g. network failure,
  5xx).

## Fix 5 ‚Äî Credential probe at startup (`startProxy`)

**Problem**: Credentials are never checked until the first `signer.sign()` call, which happens
when Claude Code sends its first real request. By then Claude Code's timeout countdown has
started. A startup probe gives the user an immediate, actionable error message before any
requests arrive.

**Solution**: Fire a non-blocking credential probe in `startProxy`. If it fails, log a prominent
error (but don't exit ‚Äî the user might refresh credentials and reconnect).

### New function (place after `createSigner`)
```typescript
export async function probeCredentials(
  config: ProxyConfig,
  timeoutMs = 5_000,
): Promise<void> {
  const provider = fromNodeProviderChain();
  let timer: ReturnType<typeof setTimeout>;
  const timeout = new Promise<never>((_, reject) => {
    timer = setTimeout(
      () => reject(new Error(`timed out after ${timeoutMs}ms`)),
      timeoutMs,
    );
  });
  try {
    await Promise.race([provider(), timeout]);
    log('INFO', `credentials OK (profile: ${process.env.AWS_PROFILE ?? 'default'})`);
  } catch (err) {
    log('ERROR',
      `credential probe failed ‚Äî AWS requests will fail until credentials are refreshed.\n` +
      `  Run: aws sso login --profile ${process.env.AWS_PROFILE ?? '<profile>'}\n` +
      `  Details: ${err}`);
  } finally {
    clearTimeout(timer!);
  }
}
```

### Change to `startProxy`
After `createSigner`, fire the probe as a background promise (do not await):
```typescript
export function startProxy(): void {
  const config = validateEnv();
  const signer = createSigner(config);

  // Non-blocking credential check ‚Äî surfaces expired SSO immediately.
  probeCredentials(config).catch(() => { /* already logged inside */ });

  // ... rest of startProxy unchanged
}
```

### Tests to add
- `probeCredentials: logs INFO when credentials resolve successfully`
- `probeCredentials: logs ERROR when provider throws`
- `probeCredentials: logs ERROR when provider times out`
- Integration: `credential probe fires on startup (INFO line appears in stderr)`

## Fix 6 ‚Äî Hardcoded version in warm-up initialize body

**Problem**: `warmBackend` sends a `clientInfo.version: "0.4.0"` hardcoded string in the
warm-up initialize request. The backend receives stale version info and log correlation is wrong.

**Solution**: One-line fix.

### Change to `warmBackend`
```typescript
clientInfo: { name: 'mcp-sigv4-proxy', version: PROXY_VERSION },
```

Replace the hardcoded `'0.4.0'` with `PROXY_VERSION` (already imported at file top).
